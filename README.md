# Pakistan Regulatory Wiki — Claude Code Plugin

A self-contained Claude Code plugin shipping the Pakistan lender regulatory knowledge base — 115 regulations (SBP, SECP, FMU, NADRA, MoITT, PTA), 39 cross-cutting concepts, 20 regulator/entity pages, and 3,217 source-cited chunks with page-accurate deep links.

Built for fintech product managers, engineers and compliance teams who need to validate PRDs / specs / deployments against Pakistan banking, microfinance, NBFC, digital-lending, EMI, payment-system and AML/CFT rules — **without leaving Claude Code**.

## What the plugin gives you

Three slash commands and three skills, all backed by an offline knowledge graph:

| Slash command | Skill | Use it when… |
|---|---|---|
| `/reg-search <question>` | `pakistan-regs-search` | You have a question — *"what does SBP say about contact-list access?"* |
| `/reg-validate <PRD or paste>` | `pakistan-regs-validate-prd` | You have a draft and want compliance flags before ship |
| `/reg-cite <doc_id or topic>` | `pakistan-regs-cite` | You need a publishable citation block for a memo / Confluence / PRD footnote |

Every answer cites with `[[doc_id]] · section · #page=N PDF link · chunk_id`. Citations are mechanically verifiable — the downstream compliance pipeline can resolve every `chunk_id` against `data/chunks.jsonl`.

## Install

### Option 1 — Local install (single user / first try)

```bash
# Clone or copy the plugin folder anywhere you like
git clone <repo-url> ~/claude-plugins/pakistan-regs-plugin
cd ~/claude-plugins/pakistan-regs-plugin

# From inside Claude Code
/plugin install ~/claude-plugins/pakistan-regs-plugin
```

### Option 2 — Private GitHub install (recommended for teams)

```bash
# As maintainer (one time)
gh repo create your-org/pakistan-regs-plugin --private --source=. --push

# As teammate (one time)
/plugin install your-org/pakistan-regs-plugin
```

Teammates authenticate with their own GitHub credentials (gh/SSO). The plugin files live on their machine; no server, no laptop dependency on the maintainer.

### Option 3 — Internal git server (Bitbucket / GitLab / Gitea)

Same as Option 2, pointed at your internal host:

```bash
/plugin install https://git.internal.example.com/your-org/pakistan-regs-plugin
```

## First-run sanity check

After install, try:

```
/reg-search Is pre-disbursement eCIB check mandatory for an MFB making a Rs. 50k consumer loan?
```

You should see an answer with:
- A direct yes/no statement
- Citations like `[[SBP_eCIB_Master]]` and `[[SBP_PRs_MFB]]`
- A clickable `#page=N` PDF link
- A `chunk_id` that resolves back to `data/chunks.jsonl`

## Update flow

```bash
cd ~/claude-plugins/pakistan-regs-plugin    # or wherever you cloned
git pull
# From inside Claude Code
/plugin update pakistan-regs
```

Changes flow when teammates explicitly re-pull. No silent updates.

## What the plugin does *not* do

- **Does not call out to any server.** Everything runs locally inside Claude Code on the teammate's machine.
- **Does not consume the maintainer's API key.** Each user pays for their own Claude usage.
- **Does not auto-update.** Updates require an explicit `git pull` + `/plugin update`.
- **Does not modify `raw/` (the source PDF cache).** That cache lives in the source repo, not the plugin — kept out for size.

## Knowledge-base provenance

Built from two source documents:
1. `Pakistan_Regulatory_Compliance_Knowledge_Base.xlsx` — 70 master regulations with regulator, category, applicable-to, URLs.
2. `Pakistan Lender Regulatory Knowledge Compilation.docx` — 49 additional URLs cited in a narrative compilation.

Layered into 115 enriched wiki pages with hand-curated Purpose / Key Sections / Applicable To, auto-detected concept and entity wikilinks, and a knowledge graph generated by re-parsing every page.

See `data/wiki/log.md` for the changelog (build history, fixes, lint passes).

## For agentic compliance pipelines

If your team's PRD-review agent or deployment validator wants to consume this KB programmatically rather than through Claude Code:

- **Vector search**: load `data/chunks.jsonl` into pgvector / Pinecone / Qdrant / Weaviate. Each chunk carries `doc_id`, `regulator`, `section_anchor`, `source_url_with_page`, `chunk_id`.
- **Graph queries**: load `data/graph.json` into Neo4j or query it as flat arrays. `applies_to` edges power "what rules apply to entity X" queries; `addresses` power "what regulations address concept Y".
- **Schema reference**: `data/wiki/CLAUDE.md` is the canonical spec — page conventions, citation format, lint rules.

The same plugin folder serves both human Claude Code users and your downstream pipeline — one artefact, two consumers.

## Quality gate

The wiki passes a 14-rule lint check (see `lint.py` in the source repo): frontmatter completeness, wikilink integrity, supersedes symmetry, alias uniqueness, controlled vocab, graph integrity, chunks coverage, citation metadata, dead-URL handling. Run `py -3 lint.py` from the source repo to verify before each release.

## Disclaimer

This plugin is an **internal compliance assistance tool**. It is not legal advice. Always verify against the official source PDF (the `#page=N` link in every citation opens the regulator's authoritative document). Where the wiki marks a source as `status: stale-link`, the curated content is still authoritative but the URL needs refreshing.

## Contributing

Edits go through the source repo, not the plugin:

```
1. Edit wiki/<doc_id>.md  (or add a new regulation)
2. Run py -3 lint.py
3. If green, run py -3 _kg_rebuild.py to refresh graph + chunks
4. git commit && git push
5. Bump version in plugin.json
6. Teammates run /plugin update
```

CLAUDE.md (in `data/wiki/CLAUDE.md`) describes the full ingest / query / lint workflows. Keep it as the source of truth for conventions.
